DocAgent — Kurulum ve Kullanım Kılavuzu

Bu kılavuz DocAgent'ı sıfırdan kurmaktan ilk dokümanın Confluence'a yayımlanmasına kadar gereken her adımı öncelik ve işlem sırasına göre anlatır. Her bölüm bir önceki tamamlandıktan sonra yapılacak adımları içerir. Kurulumun tamamı 20–30 dakika sürer.

1. Genel Bakış

DocAgent, bir web uygulamasının ekranlarını Playwright ile otomatik tarayıp; BRD, Confluence space'leri, Jira projeleri, Swagger spec'leri ve örnek kılavuz şablonlarıyla harmanlayarak her ekran için iki Türkçe doküman üreten yapay zeka destekli bir masaüstü araçtır:

  • 📘 Kullanıcı Kılavuzu — son kullanıcıya yönelik, adım adım, ekran görüntüleri gömülü
  • ⚙️ Teknik Döküman — geliştiriciye yönelik, API endpoint'leri + iş kurallarıyla ilişkilendirilmiş
Akış özeti Hedef URL → Ekran keşfi (Playwright) → Bağlam (RAG) → Claude analiz → Kullanıcı kılavuzu + Teknik doc → Kapsam doğrulama → Confluence'a yayım veya DOCX/MD/PDF/ZIP indirme.

Çekirdek özellikler:

  • Etkileşimli keşif — sadece statik sayfa değil; filtre paneli, modallar (Ekle/Düzenle/Detay/Log), satır aksiyonları öncelik sırasıyla yakalanır
  • Çok kaynaklı RAG — BRD, Confluence, Jira, Swagger, şablon aynı bağlamda; çift okuma engelli, her kaynak tipi için bütçe garantili. Türkçe çekim eklerine toleranslı tokenizer.
  • Kapsam doğrulama — her UI öğesinin dokümanda geçtiği kontrol edilir + Haiku LLM-judge "anlamlı anlatılmış mı?" sorusuyla sahte coverage'ı düşürür; eksik varsa Claude'a hedefli düzeltme turu attırılır (eşik + tur sayısı .env'den ayarlanır)
  • Hedef uygulama bağımsızlığı — sidebar/global nav tespitini LLM her ekranda yeniden karar verir (eski hardcoded liste yalnız legacy fallback)
  • Heading'siz .docx/.pdf parser — numbered "1.1." ve ALL-CAPS "AMAÇ" sezgisi ile bölümlere ayırır (eski tek-blok davranışı yok)
  • Partial retry — başarısız job'larda yalnız eksik ekranlar için yeniden üretim, tamamlananlar yeniden ödenmez
  • Prompt caching — aynı job içinde 2. ekrandan itibaren input token'ların büyük kısmı Anthropic ephemeral cache'inden gelir (~%90 ucuz)
  • İki Claude backend — Claude Code CLI veya Anthropic SDK
  • Yerel depolama — tüm referans & çıktı yerel diskte; ekibe açık sunucu yok
  • Güvenlik — yerel API CSRF guard (X-DocAgent header), CORS localhost-only, settings allowlist + newline injection guard, .env 0o600 (sahip-only), upload 25MB + uzantı allowlist

2.5 .env dosyası

5

Asgari yapılandırma

Proje kökünde .env oluşturun:

cat > .env <<'EOF'
PORT=3000
CLAUDE_BACKEND=cli          # 'cli' veya 'api'
DOC_LANGUAGE=tr
MAX_DISCOVERY_DEPTH=0       # 0 → tek ekran modu (etkileşimli keşif)

# Coverage fix-up — üretilen doküman bu eşiğin altındaysa eksik UI
# öğeleri için ek LLM turu tetiklenir.
FIX_UP_THRESHOLD=90         # 0-100; düşürünce hız ↑ maliyet ↓ kapsam ↓
FIX_UP_MAX_PASSES=2         # 0-5; 0 → fix-up tamamen kapalı

# Coverage LLM-judge — substring match'in "label var = açıklanmış" gibi
# yalancı pozitiflerini Haiku ile doğrular ("anlamlı şekilde anlatılmış
# mı?"). Fix-up turu daha doğru hedeflere yönelir. Maliyet ~$0.005/ekran.
COVERAGE_LLM_JUDGE=true     # 'false' → saf substring match

# Hedef uygulama login arkasındaysa:
# APP_BASE_URL=https://...
# APP_USERNAME=...
# APP_PASSWORD=...

# Canlı Uygulama Kanıtı (MCP) — opt-in; bkz. Bölüm 6.6
# LIVE_APP_MCP_ENABLED=false      # 'true' → gerçek ekran/network gözlemi
# LIVE_APP_MCP_TIMEOUT_MS=480000  # tarayıcı gezinme zaman aşımı (8 dk)

# API modu kullanılacaksa:
# ANTHROPIC_API_KEY=sk-ant-api03-...

# Atlassian OAuth (arayüzden de doldurulabilir, Adım 3.3'e bakın):
# ATLASSIAN_OAUTH_CLIENT_ID=
# ATLASSIAN_OAUTH_CLIENT_SECRET=
EOF
chmod 600 .env
Güvenlik .env dosyası .gitignore'da, commit edilmez. Sunucu her .env yazımında otomatik 0o600 moduna ayarlar (sahip-only); manuel chmod 600 yine de ilk oluşturmada faydalıdır. ATLASSIAN_* ve ANTHROPIC_API_KEY gibi sırları ekip arkadaşlarınızla şifreli kanal dışında paylaşmayın.
Lazy okuma env.ts her okumada güncel process.env'i döndürür — Ayarlar arayüzünden değişiklik yapınca sunucu restart'a gerek kalmaz.
Settings yazma allowlist POST /api/settings yalnız tanınan key'leri kabul eder (CLAUDE_*, APP_*, ATLASSIAN_OAUTH_*, CONFLUENCE_*, PORT, DOC_LANGUAGE, MAX_DISCOVERY_DEPTH, FIX_UP_*). Allowlist dışı key'ler sessizce atılır + sunucu log'una uyarı düşer. ATLASSIAN_ACCESS_TOKEN/REFRESH_TOKEN gibi OAuth-yazılı alanları manuel ayarlamayın — OAuth flow kendi yazar/yeniler.

3.3 Atlassian OAuth opsiyonel

Confluence space / sayfa veya Jira projelerini bağlam olarak kullanmak ve Confluence'a yayımlamak için OAuth bağlantısı gerekir. Klasik Confluence v1 API HTTP 410 Gone döndürdüğü için DocAgent yalnızca v2 API + granular OAuth scope'ları kullanır.

1

Atlassian Developer Console'da uygulama oluştur (bir kez)

developer.atlassian.com/console/myapps → Create → OAuth 2.0 integration.

Permissions'a şu scope'ları ekleyin (tam liste):

read:space:confluence
read:page:confluence
write:page:confluence
read:attachment:confluence
write:attachment:confluence
read:content-details:confluence
read:jira-work
write:jira-work
offline_access

Authorization → Callback URL:

http://localhost:3000/api/auth/atlassian/callback
PORT'u değiştirdiyseniz .env'de PORT=3000 dışında bir değer verdiyseniz, Atlassian Developer Console'da callback URL'sini de aynı port'a güncelleyin (örn. http://localhost:4000/api/auth/atlassian/callback). Aksi halde OAuth flow son adımda redirect_uri_mismatch ile patlar. DocAgent'ın getRedirectUri()'si zaten env.port'dan türetir; eşleşme Atlassian tarafındadır.
Ekip kullanımı Tek bir Atlassian uygulaması oluşturup Client ID + Secret'ı ekiple paylaşabilirsiniz (her analist kendi .env'sine koyar). Token'lar her makinada ayrı yazılır, paylaşılmaz.
2

Client ID + Secret'ı Ayarlar'a yapıştır

Ayarlar → Atlassian (Confluence + Jira): OAuth Client ID + Secret → Kaydet.

3

Bağlan

Atlassian'a Bağlan → tarayıcı Atlassian onay ekranına yönlenir → siteyi seçin, izinleri onaylayın → DocAgent'a geri döner. Token'lar .env'e yazılır (ATLASSIAN_ACCESS_TOKEN / REFRESH_TOKEN / EXPIRES_AT / CLOUD_ID / SITE_URL / SCOPE).

4

Test

Aynı bölümde Test butonu — yeşil "OK" görmelisiniz. Hata alıyorsanız Sorun Giderme bölümüne bakın.

Scope güncellemesi Daha sonra OAuth uygulamasına scope ekler/çıkarırsanız, mevcut token yeni scope'ları içermez. Çözüm: Bağlantıyı Kes → Yeniden Bağlan.

4. Referansları Yükle

Temiz kurulum — hazır örnek veri yoktur Yeni bir git clone sonrası data/ klasörü boştur (gizlilik gereği örnek iş verileri repoya konmaz). Kendi BRD, Confluence, Swagger ve şablonlarınızı bu bölümdeki adımlarla yüklemeniz gerekir — aksi halde üretilen doküman yalnız ekran görsellerine dayanır ve iş kuralı bağlamından yoksun kalır.

4.1 Hangi sırayla yüklemeli?

Üretilen dokümanın kalitesi tamamen referansların kalitesine bağlıdır. Doküman üretmeden ÖNCE aşağıdaki sırayla yükleyin:

#KaynakNeden öncelikli?
1BRD doküman(lar)ı (.docx/.pdf) RAG'da en yüksek kaynak önceliği (1.0) — iş kuralının ana kaynağı
2Confluence space (varsa) veya tek sayfalar Süreç anlatımı, takım jargonu (öncelik 0.85)
3Swagger / OpenAPI API endpoint'leri teknik dökümana eşleştirilir
4Jira projesi Tarihsel kararlar, ticket geçmişi (öncelik 0.75)
5Örnek kılavuz şablonu Üretilen kılavuzun üslubu ve formatı buna benzetilir
Kaynak önceliği nasıl çalışır? DocAgent retrieval skorunu kaynak tipine göre çarpar: BRD ×1.0, Süreç Analizi ×0.95, Confluence ×0.85, Jira ×0.75, Manuel ×0.6. Yüklenen reference tipi doc'lar otomatik Süreç Analizi olarak etiketlenir (BRD tipi seçilmezse). Ayrıca her kaynak tipinden en az 2 bölümün prompt'a girmesi garanti altına alınır (balanceBySourceType), yani BRD bağlam bütçesini tek başına yutmaz.
Heading'siz .docx / .pdf de bölümlere ayrılır mammoth (.docx) ve pdf-parse (.pdf) çıktısı düz metin olarak gelir; DocAgent sezgisel heading tespiti yapar: numbered outline (1., 1.1., 1.1.1.) ve standalone ALL-CAPS (AMAÇ, GEREKSİNİMLER) başlık sayılır. Ölçüm: 76 sayfalık bir spec PDF'i 1 → 52 anlamlı bölüme ayrıldı. Markdown # heading'li doc'larda eski parser kullanılır.

4.2 Veri Kaynakları (toplu Confluence / Jira)

Sol menüden Referanslar → ilk sekme Veri Kaynakları.

1

Confluence Space ekle

Space Key girin (örn. DOCS) — Atlassian space URL'sinde /spaces/DOCS/ kısmı. Opsiyonel açıklama → Ekle.

2

Jira Projesi ekle

Proje Key girin (örn. MBS veya PROJ). Otomatik büyük harfe çevrilir. Ekle.

3

Senkronize Et

Her satırın yanındaki Senkronize Et butonu o kaynağı çeker:

  • Confluence space → her sayfa data/references/confluence/<pageId>.txt olarak kaydedilir (Confluence v2 API, body-format=storage + entity decode + PDF-style TOC/header temizliği).
  • Jira projesi → aktif/tamamlanmış statüdeki issue'lar tek dosyada data/references/jira/<KEY>.json (ADF description'lar düz metne çevrilir).
Jira statü filtresi Henüz başlanmamış / iptal işler bağlama alınmaz: Backlog, To Do, Cancel / İptal (ve Türkçe/İngilizce varyasyonları) statüsündeki issue'lar atlanır — bunlar kararlaştırılmış iş değildir ve üretilen kılavuzu yanıltabilir. In Progress, In Review, Done gibi tüm diğer statüler baz alınır. Senkronizasyon log'unda kaç issue'nun bu nedenle atlandığı görünür. Statü ismi kurumunuzda farklıysa src/ingestion/jiraStatusFilter.ts içindeki listeye eklenebilir.

Üstte Tümünü Güncelle butonu kayıtlı tüm kaynakları sırayla senkronize eder.

İpucu Senkron büyük space'lerde 1-2 dk sürebilir. Tek seferlik; veriler değişince yeniden çalıştırın.

4.3 Tek Confluence sayfası

Confluence sekmesi. Sayfa URL'sini yapıştırın → Ekle. Sadece o sayfa çekilir (URL'den pageId çıkarılır, içerik storage formatta indirilir).

4.4 Swagger / API

Swagger / API sekmesi. Üç alan var:

  • Servis Adı: liste için ad (örn. payment-service)
  • Swagger URL: OpenAPI JSON URL'si
  • Authorization opsiyonel — korumalı spec için Bearer ... veya Basic ... başlığı
  • SSL doğrulamasını atla — self-signed iç sertifikalı sunucular için

Çekilen spec data/swagger/ altına timestamp'li bir JSON olarak iner; endpoint sayısı listede gösterilir.

4.5 BRD & referans dökümanlar

Dökümanlar sekmesi. .docx / .pdf / .md / .txt yüklenebilir.

  • Tip: BRD veya Genel Referans
  • Kaynak / Açıklama: liste için meta veri

Yükleme sırasında:

  • .docx → mammoth.extractRawText
  • .pdf → pdf-parse (PDFParse.getText)
  • İçerik cleanReferenceText ile temizlenir: TOC dot-leader, sayfa numarası, tekrar eden header/footer ayıklanır
PDF kalitesi Taranmış / OCR'sız PDF'ler okunamaz — kaynak olarak işlevsizdir. Metin tabanlı PDF kullanın veya önce OCR'dan geçirin.

4.6 Örnek kılavuz şablonu

Şablonlar sekmesi. Daha önce yazılmış örnek kullanıcı kılavuzlarını yükleyin. DocAgent bunların ilk 7000 karakterini üslup ve format referansı olarak Claude'a verir (paragraf yapısı, başlık tarzı, hitap tonu).

İçerik vs. format ayrımı Şablonun içeriği KOPYALANMAZ — sadece anlatım üslubu, adım numaralandırma, "sen/siz" hitabı taklit edilir. Bu kural prompt'ta açıkça yazılı.

5.2 Ekran keşfi

1

URL gir

Ekran Keşfi → Hedef Uygulama alanına taranacak ekranın direkt URL'sini girin (örn. https://app.firma.com/event-management/prematch-program). İsterseniz Ek URL butonuyla birden fazla ekran kuyruğa alabilirsiniz.

2

Bağlam Filtresi opsiyonel

Referans havuzunuz çok büyükse (yüzlerce Confluence sayfası, binlerce Jira issue'su) Anahtar Kelimeler ve Confluence Sayfa Adları alanlarıyla yalnızca konuyla ilgili olanları bağlama dahil edebilirsiniz. Boş bırakırsanız tüm ilgili referanslar skoru > 0 olduğu sürece değerlendirilir.

3

Başlat

Başlat → arka planda bir discovery job'ı tetiklenir. İlerleme SSE üzerinden canlı akar:

  • Tarayıcı başlatılıyor → giriş yapılıyor (env'deki USERNAME/PASSWORD ile)
  • Ana ekran yükleniyor → analiz
  • Etkileşimli keşif (sırayla):
    • Filtre paneli pre-pass (13 farklı selector)
    • Sekmeler (≤6), açılır menüler (≤5)
    • Kolon header'ları → satır aksiyonları → satır-edit ikonları
    • Aksiyon butonları (öncelik: add, yeni, ekle, edit, düzenle, filter, log, detay, info, manual — toplam ≤10 modal)
    • Tarih seçiciler (≤4), checkbox, toggle

Süre: tek ekran için 30-90 saniye. Bir ekran için 20-30 state görüntüsü tipiktir.

4

Ekranları seç

Keşfedilen ekranlar listesinden dokümante edileceklerini işaretle (veya "Tümünü Seç"). Her satır: başlık + ana görsel + state sayısı.

Login arkası ekran Hedef uygulama login ardındaysa .env'de APP_BASE_URL / APP_USERNAME / APP_PASSWORD ayarlı olmalı. SSO/cookie tabanlı sistemlerde Playwright otomatik giriş zayıf — geliştirme/test ortamı tercih edin.

5.3 Doküman üretimi

1

Derin Analiz seçimi

Üst çubukta Derin Analiz anahtarı:

  • Kapalı (varsayılan) — hızlı, ucuz, çoğu ekran için yeterli
  • Açık — Claude'a daha derin akıl-yürütme talimatı; karmaşık ekran ve süreçler için
2

Üretimi başlat

Seçili ekranlarla Doküman Üret. İlerleme sayfası açılır.

Arka planda her ekran için sırayla:

  1. Bağlam yükleme — referans envanteri loglanır (örn. "Section types: {brd:4,confluence:2,jira_task:3}")
  2. Ekran analizi — Claude görselleri okur, ana içerikteki tüm UI öğelerini (buton, alan, filtre, kolon, sekme, modal tetikleyici + açık modal/panel görsellerindeki iç alanlar) eksiksiz listeler + iş akışlarını çıkarır. 8000 token tavanı ~100+ öğeye yeter; analiz çıktısı tavana takılırsa ekran failed olur (sessiz eksik yerine görünür hata) ve Geçmiş → ⟳ Eksikleri Üret ile tekrar denenir. Sidebar/global nav öğeleri bu ekranın parçası sayılmaz, dışlanır.
  3. RAG seçimi — ilgili bölümler skor ×kaynak önceliği ile sıralanır; 16KB bütçeye, her kaynak tipinden en az 2 bölüm garantili
  4. İki paralel üretim: Kullanıcı Kılavuzu + Teknik Döküman (her biri varsayılan 8000 token tavanı + image-dolu prompt). Çıktı bu tavana takılırsa doküman footer'ında ⚠️ Çıktı kesildi uyarısı görünür.
  5. Prompt caching — rol + output structure + kurallar + şablon bloğu job içinde sabit; 2. ekrandan itibaren Anthropic ephemeral cache hit verir (input token ~%10'a düşer). Konsolda [claude] cache: read=… create=… olarak loglanır.
  6. Kapsam kontrolü — her UI öğesinin (sidebar nav hariç) dokümanda geçtiği fuzzy-match ile sayılır. Hangisi "sidebar nav" sayılır? LLM (analyzeScreen) her öğeye isGlobalNav: true|false ataması yapar — hedef uygulamadan bağımsız. Eski/cached analizler için hardcoded hint listesi fallback.
  7. LLM-judge doğrulaması (COVERAGE_LLM_JUDGE=true, default açık) — substring "label var" der ama "anlamlı anlatılmış" garanti etmez. Haiku 4.5 her covered öğeye yes/no döner; "anlatılmadı" denenler missing'e geri taşınır → fix-up doğru hedefe yönelir. Hata olursa raw coverage'a graceful fallback (regresyon yok). Maliyet ~$0.005/ekran.
  8. Düzeltme turu — kapsam < FIX_UP_THRESHOLD (varsayılan %90) ise eksikler için en fazla FIX_UP_MAX_PASSES (varsayılan 2) turlu hedefli fix-up; her tur regresyon yapmıyorsa kabul. Kapsam aynı kalsa bile eksik set'i değiştiyse bir sonraki tur denenir (farklı eksiklere fırsat). İkisi de .env'den ayarlanır.
  9. Doküman sonuna Üretim Bilgisi izi eklenir

Job paralel 3 ekran işler (CONCURRENCY=3). Tipik süre: ekran başına 1–3 dk; 10 ekranlık iş ~15-30 dk.

3

İlerlemeyi izle

İlerleme sayfasında SSE ile canlı log + her ekran için "✓" işareti görürsünüz. Kontroller:

  • Duraklat — checkpoint'te bekler (cancel'lık değil)
  • Devam — duraklatılmış işi sürdürür
  • İptal — soft cancel (bir sonraki checkpoint'te durur); tamamlanmış ekranların dökümanları korunur
"Prompt too long" hatası Otomatik 3 kademeli geri çekilme: tam bağlam (7000 char şablon) → yarı state (3500 char şablon) → 4 state (1500 char şablon). Manuel müdahale gerekmez. Çok sık alıyorsanız bağlam filtresi (6.1) kullanın.
Job watchdog (idle-tabanlı) Eski 30 dk sabit eşik kaldırıldı; artık 3 saat hareketsizlik (ekran üretim güncellemesi yok) veya 12 saat toplam süre tetiklenirse job failed işaretlenir. 50-80 ekranlık büyük üretimler sorunsuz tamamlanır.
Bir job yarıda kalırsa — token kaybı yok Tamamlanmış ekranların dokümanları her zaman Dökümanlar sayfasında durur (per-ekran kalıcı yazım). Eksiklerini üretmek için: Geçmiş sayfasında failed/cancelled satırının altındaki "⟳ Eksikleri Üret (N)" düğmesi tamamlananları yeniden ödemeden yalnız eksik ekranlar için yeni bir job başlatır.

6. İleri Düzey

6.1 Bağlam filtresi

Büyük referans havuzlarında (yüzlerce sayfa, binlerce issue) her analizde tüm bağlamı taramak hem yavaş hem maliyetli olabilir. Ekran Keşfi sayfasındaki Bağlam Filtresi ile odaklayın:

  • Anahtar Kelimeler: "ödeme, tahsilat" → bu kelimeleri içeren bölümler önceliklenir
  • Confluence Sayfa Adları: başlığın bir kısmı → sadece o sayfalar

Filtre yokken bile sistem zaten skor=0 olan (alakasız) bölümleri otomatik atar; filtre bunu sıkılaştırır.

6.2 Prompt özelleştirme deneyimli

DocAgent'ın üretim davranışı Sistem Promptları sayfasında düzenlenebilir. Anahtarlar:

  • userManual — kullanıcı kılavuzu prompt'u (role, outputStructure, instructions, rules[], maxTokens)
  • technicalDoc — teknik döküman prompt'u
  • Fix-up, bölüm rejenerasyonu, ekran analizi promptları

Yapılandırma data/prompts/config.json'da tutulur. Bozulursa dosyayı silip uygulamayı yeniden başlatmak varsayılana döner.

Dikkat Promptlar AI'nın tüm çıkartı belirler. Büyük değişiklikten önce config.json'u yedekleyin.

6.3 Derin Analiz anahtarı

Üst çubuktaki toggle. Açıkken Claude'a daha kapsamlı akıl-yürütme talimatı verir; karmaşık iş kuralları, çok sayfalı modallar veya çapraz referans gereken ekranlarda kullanın. Kapalı modda hız ve maliyet öne çıkar.

6.4 Geçmiş

Geçmiş sayfası tüm job kayıtlarını listeler (keşif + üretim). Her satır: tarih, durum (completed/failed/ cancelled), süre, ekran sayısı, üretilen doküman sayısı. data/db/jobs.json'da saklanır.

⟳ Eksikleri Üret düğmesi Bir doküman job'ı başarısız veya iptal olduysa, satırda eksik ekran sayısı ("3/25 eksik" gibi) görünür ve ⟳ Eksikleri Üret düğmesi çıkar. Tıklayınca yalnız üretilmemiş ekranlar için yeni bir job başlar; tamamlananlar yeniden ödenmez (token tasarrufu). Eski sürümden kalma job'larda bu bilgi olmadığı için düğme görünmez — o durumda Ekran Keşfi'nden manuel yeniden başlatın.

6.5 Güncelleme

İki yol var:

  • Arayüzden: Güncelleme sayfası → Bilgi ile uzaktaki son sürümü gör → Güncelle → uygulama scripts/update.sh'yi çalıştırır; git reset --hard origin/main, bağımlılık güncellemesi, client build ve servis yeniden başlatması yapar. Loglar data/logs/update.log'a yazılır.
  • Terminal: 
    cd ~/projects/documentation-agent
git pull
npm install                  # gerekirse
(cd client && npm install)
npm test                     # tüm testler yeşil olmalı
Lokal değişiklikleriniz Arayüz güncellemesi git reset --hard kullanır — .gitignore dışındaki commit edilmemiş değişiklikleriniz kaybolur. Terminal'den önce git status ile kontrol edin; lokal değişikliğiniz varsa commit veya git stash ile saklayın. .env, data/ ve node_modules/ ignore'da olduğu için güvendedir.
Yeniden başlatma Sunucu kodu değişimi için uygulamayı tamamen kapatıp yeniden açın (sekmeyi yenilemek yetmez — sunucu süreci hâlâ eski kodla çalışır). scripts/update.sh bunu zaten otomatik yapar.

6.6 Canlı Uygulama Kanıtı (MCP)

Normalde DocAgent ekranları Playwright ile gezip ekran görüntüsü yakalar ve kılavuzu bu görsellerden + referanslardan üretir. Canlı Uygulama Kanıtı özelliği bunu değiştirmez; üstüne ek bir katman ekler: her ekran için Claude, bir Playwright MCP sunucusu üzerinden gerçek bir tarayıcıyı kendisi sürer, sekmeleri ve CRUD akışlarını (ekle/düzenle/sil/görüntüle) gerçekten dener, tetiklenen ağ (network) isteklerini ve doğrulama/hata mesajlarını gözlemler. Bu gözlem, kılavuz üretimine en güvenilir kaynak olarak beslenir (statik BRD ile çelişirse gerçek gözlem tercih edilir).

Varsayılan KAPALI (opt-in) Bu özellik Claude'un ekranı tek tek gezmesini gerektirdiğinden belirgin biçimde daha yavaş ve daha pahalıdır (ekran başına çok sayıda ek adım). Yalnızca en yüksek doğruluğun gerektiği durumlarda açın. Açmak için: Ayarlar → Canlı Uygulama Kanıtı (MCP) bölümündeki kutuyu işaretleyip Kaydet'e basın (eşdeğeri .env: LIVE_APP_MCP_ENABLED=true).

Gereksinimler:

  • CLAUDE_BACKEND=cli (özellik yalnız CLI backend'de çalışır).
  • Node.js/npm kurulu ve PATH'te — MCP sunucusu npx @playwright/mcp ile başlatılır.
  • APP_BASE_URL tanımlı (gezilecek hedef). Giriş gerekiyorsa APP_USERNAME/APP_PASSWORD ile otomatik giriş yapılır — ek manuel adım yok.
Hazır mı? — önkontrol Ayarlar sayfasındaki bölüm, hazır-olma durumunu (CLI backend / npx / Uygulama URL / Otomatik giriş) otomatik gösterir; hazır değilse eksik olanı Türkçe açıklar. Aynı bilgiyi API'den GET /api/live-app/status da verir: { ready, npx, backend, appUrlSet, autoLogin, reason }. ready:false ise reason alanı eksik olanı Türkçe ve eyleme dönük anlatır (ör. "npx bulunamadı…", "CLAUDE_BACKEND=cli yapın…").
Güvenlik Tarayıcı araçları dar bir allowlist ile sınırlıdır (gezinme, tıklama, form doldurma, ağ/DOM gözlemi) — keyfi JavaScript çalıştırma (browser_evaluate) ve dosya yükleme kapsam dışıdır. Giriş şifresi Claude'a komut satırı argümanı değil, stdin üzerinden verilir (ps çıktısında görünmez) ve hata-ayıklama prompt dökümünde maskelenir. Oturum çerezleri data/.live-app-profile/ (gitignore'da) kalıcı profilinde tutulur.
Güvenli varsayılan (fail-safe) npx yoksa, zaman aşımı olursa veya MCP hata verirse özellik sessizce atlanır — kılavuz üretimi kanıtsız, normal şekilde devam eder. Toplanan kanıt ekran başına data/references/live-app/ altında önbelleğe alınır; "Eksikleri Üret" veya yeniden üretim aynı ekran için tekrar ödeme yapmaz.

6.7 Adım Vurgusu, Bulanıklaştırma, Stil Denetimi

Üç görsel/kalite seçeneği Ayarlar → Hedef Uygulama → Keşif ve görsel seçenekleri altından açılıp kapatılır:

  • Adım vurgusu (varsayılan AÇIK): Keşif sırasında bir buton/satır işlemi tıklanmadan hemen önce öğenin etrafına kırmızı çerçeve + rozet çizilir ve o anın görüntüsü "Adım: … (konumu işaretli)" olarak yakalanır. Kılavuz "X'e tıklayın" derken butonun ekranda nerede olduğunu gösteren görseli de sunar (Scribe/Tango tarzı). Kapatmak: ANNOTATE_STEPS=false.
  • Hassas veri bulanıklaştırma (varsayılan KAPALI): Görüntü alınmadan önce olası kişisel veriler (e-posta, telefon, TCKN, IBAN içeren öğeler) CSS ile bulanıklaştırılır. Regex tabanlı en-iyi-çaba — kurumsal paylaşım öncesi görselleri yine gözden geçirin. Açmak: REDACT_SENSITIVE=true.
  • Stil denetimi (varsayılan AÇIK): Üretim sonrası ucuz bir modelle yalnız biçimsel düzeltme yapılır: UI adları kalınlaştırılır, adım numaraları sıralanır, İngilizce jargon Türkçeleştirilir. İçerik değişmez — uzunluk/görsel sayısı sapan çıktı otomatik reddedilir. Kapatmak: STYLE_LINT=false.

7. Yaşam Döngüsü / Kapatma

DocAgent yerel masaüstü uygulamasıdır. Yaşam döngüsü kuralları:

EylemSonuç
Sekmeyi kapat (tek sekme açıksa) pagehide → 30sn grace → reconnect yoksa process.exit(0) → supervisor vite'ı kapatır
⌘+R veya ⌘+Shift+R ile yenile pagehide tetiklenir ama yeniden yüklenen sayfa 1-3sn'de heartbeat gönderip grace sayacını iptal eder → uygulama açık kalır
Sekmeyi arka plana al pagehide tetiklenmez. Heartbeat throttle olur ama 15 dk fallback hiç tetiklenmez → uygulama açık kalır
Tarayıcı çöker / makineyi uyut 15 dakika heartbeat yoksa crash fallback tetiklenir ve sunucu kapanır (kalıntı süreç bırakmamak için)
Birden fazla sekme açık, birini kapat Diğer sekmenin heartbeat'i grace sayacını iptal eder → açık kalır

Yeniden açmak için: npm run dev ya da masaüstü ikonu.

Çözüm: DocAgent sekmesini yenileyin; API'yi dışarıdan çağırıyorsanız -H "X-DocAgent: 1" ekleyin.

Ayarlar'da bir key kaydedilmedi Allowlist dışı bir key gönderdiniz; sunucu log'unda [settings] allowlist dışı/geçersiz key'ler atıldı: … uyarısı görürsünüz. Yalnız tanınan key'ler yazılır (bkz. 2.5 — Settings yazma allowlist kutusu).

9. Sıkça Sorulanlar

Veriler buluta gönderilir mi?

Hayır. Tüm referanslar, ekran görüntüleri ve üretilen dokümanlar yerel diskte (data/) tutulur. Yalnızca Claude API çağrılarında prompt + görseller Anthropic'e gider (CLI modunda Claude Code'un veri politikası, API modunda Anthropic API politikası geçerlidir).

Hangi modeli kullanıyor?

claude-sonnet-4-6 (text + vision). Her iki backend de aynı modeli kullanır.

10 ekranlık bir iş ne kadar tutar?

Kabaca her ekran 1-3 dk + 2 doküman + olası 0-2 düzeltme turu. Tipik 10 ekran ~15-30 dk. Maliyet tarafı backend seçimine bağlı — CLI modunda Claude Code aboneliğiniz kapsamında; API modunda console.anthropic.com → Usage'da ölçülür.

Birden fazla ekran aynı anda işlenir mi?

Evet. CONCURRENCY=3 ekran paralel işlenir (her biri kendi Claude çağrılarını yapar; toplam 3x üretim teorik). Daha fazla paralel için src/server/jobs/documentationJob.ts'i düzenleyebilirsiniz, ancak Claude rate-limit'ine dikkat.

Bir doküman ne kadar uzun olur?

Kullanıcı kılavuzu için tipik 15-25 KB markdown; 8-12 ekran görseli gömülü, ekran başına 3-5 ana iş akışı + her UI öğesi anlatımı. Teknik döküman benzer boyutta, daha az görsel.

Aynı ekranı tekrar tekrar üretebilir miyim?

Evet. Her üretim yeni bir doküman olarak Dökümanlar sayfasında durur; eski olanlar Geçmiş'ten erişilebilir. Tek bölümü yenilemek için "Bölüm Yeniden Üret" daha ucuz.

BRD ne kadar büyük olabilir?

Pratikte 100-200 sayfa BRD'ler sorunsuz işlenir. Çok büyük BRD'lerde retrieval doğal olarak yalnız ilgili bölümleri seçer; bütçe 16KB bölüm + 9 paragraf — gerisi ihtiyaç olursa ekrana özel bağlam filtresiyle daraltılabilir.

Confluence'a yayımlarken ekran görüntüleri gidiyor mu?

Evet. Yayım sırasında attachment olarak yüklenir (v1 multipart endpoint; v2'de henüz multipart yok). Attachment hatası fatal değil — sayfa yine oluşur, görseller eksik gelir; ekibe duyurun.

Doküman kalitesi düşükse?

Sırayla:

  1. BRD ve örnek şablon yüklü mü? Yoksa kalite kayda değer düşer.
  2. Üretim Bilgisi dipnotunda hangi referansların kullanıldığını inceleyin — beklediğiniz kaynak orada yoksa filtre veya ad uyumsuzluğu vardır.
  3. Tek bölümü "Bölüm Yeniden Üret" ile düzeltmeyi deneyin.
  4. Sistemli problem varsa userManual prompt'unu Sistem Promptları'ndan düzenleyin.

Multi-user / sunucu deploy mümkün mü?

Şu an hayır — DocAgent tek kullanıcılı yerel uygulama olarak tasarlandı (JSON tabanlı kalıcılık, auth katmanı yok). Yol haritasında DB + auth + sunucu deploy var.

Ekipçe nasıl çalışıyoruz? Her analist kendi kurulumunu mu yapıyor?

Evet. Her analist kendi makinesine kurar, kendi .env'sini oluşturur. Paylaşılabilenler:

  • Atlassian Developer Console'da tek bir OAuth uygulaması oluşturup Client ID + Secret ekiple paylaşın (sırlar şifreli kanaldan). Token'lar her makinada ayrı yazılır.
  • data/brd/*.md, data/swagger/*.json, data/prompts/config.json gibi repository-içi kaynaklar commit edilirse herkes aynı bağlamla çalışır.

Paylaşılmayanlar: .env (sırlar), data/db/*.json (lokal job/doküman geçmişi), data/screenshots/ (lokal keşif çıktıları), data/references/ (lokal Confluence/Jira sync'leri).

Üretilen dokümanları nasıl paylaşırız? İki yol: (1) Confluence'a yayımla — ekibin geri kalanı oradan okur, (2) Dökümanlar sayfasından ZIP olarak indir (markdown + ekran görüntüleri) → Slack/e-posta vb. ile paylaş.

API modunda aynı anda 3 analist üretim yaparsa ne olur?

Hepsi aynı ANTHROPIC_API_KEY'i paylaşıyorsa Anthropic org-bazlı rate-limit'i ortak tüketilir. Pratik öneri: her analistin kendi API key'i olsun (Anthropic Console'dan üretilebilir), veya CLI modu kullanılsın (Claude Code her makinada ayrı oturum açar; rate-limit ayrı).

Bir alan kaydetmedim ama "OK" gördüm — neden?

Allowlist dışı key gönderdiniz (örn. yazım hatasıyla ANTHROPC_API_KEY). Sunucu işlemi başarılı sayar ama o key'i sessizce atar; data/logs/server.log'da [settings] allowlist dışı/geçersiz key'ler atıldı: … uyarısı vardır. Allowlist için bkz. 2.5 — Settings yazma kutusu.

Bir job yarıda kaldı veya iptal ettim — yeniden başlatınca her şey baştan mı?

Hayır. Tamamlanmış ekranların dokümanları Dökümanlar sayfasında durur. Geçmiş sayfasında o job'un satırında ⟳ Eksikleri Üret (N) düğmesi görünür; yalnız üretilmemiş ekranlar için yeni bir job başlatır. Tamamlananlar yeniden ödenmez (önemli token tasarrufu, özellikle API modu).

Coverage %95 ama doküman gerçekten yetersiz olabilir mi?

Eski sürümde evet: substring "label var" sayardı. Şimdi COVERAGE_LLM_JUDGE=true (varsayılan açık) ile Haiku 4.5 her covered öğeyi "anlamlı anlatılmış mı?" diye doğrular; sadece etiket mention'ları missing'e geri taşınır → fix-up doğru hedefe yönelir. Ekstra maliyet ~$0.005/ekran. Kapatmak için false.

Ekrandaki her buton ve alan kılavuza giriyor mu?

Hedef bu. Üç katmanlı güvence: (1) Ekran analizi ana içerikteki tüm UI öğelerini eksiksiz listeler (8000 token tavanı, "kısaltma" yasağı; tavana takılırsa ekran failed olur, sessizce eksik kalmaz). (2) Üretim her öğeyi kılavuzda işler; kapsam eşiğin altındaysa hedefli düzeltme turu çalışır. (3) LLM-judge "sadece adı geçiyor" ile "gerçekten anlatılmış"ı ayırır. Sınır: bir öğe yalnızca bir modal/panel içinde görünüyorsa, o modalın keşifte yakalanmış olması gerekir (Ekran Keşfi modal limiti varsayılan 10). Üretim Bilgisi dipnotundaki kapsam % ve eksik listesi her zaman gerçek durumu gösterir.

Sync edilen Confluence/Jira sayfası kötü bir talimat içerirse doküman tehlikeye girer mi?

Bu bir "prompt injection" riskidir. Confluence'taki bir paragrafta "ignore previous instructions" gibi metin varsa Claude'a olduğu gibi gönderilir ve üretilen dokümanı yan etkiler. Tam koruma zor; iş süreci olarak sync ettiğiniz Confluence space ve Jira projelerinin güvenilir olmasına dikkat edin. Bilinmeyen dış kaynakları bağlamayın.
